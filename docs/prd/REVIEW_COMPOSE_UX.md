# CartPick 리뷰 작성 UX 설계

## 0. 핵심 원칙

리뷰 작성 폼은 다음 3가지 원칙만 지키면 된다:

1. **3초 안에 시작 가능해야 한다**
2. **생각 없이도 쓸 수 있어야 한다**
3. **쓰는 게 "부담"이 아니라 "반응"이어야 한다**

> 리뷰 = 글쓰기 ❌
> **리뷰 = 반응** ⭕

---

## 1. 전체 플로우 (단일 화면)

```
[상품 상세]
    ↓ ⭐ 클릭 or 리뷰쓰기 버튼
    ↓
┌─────────────────────────┐
│  × 닫기                  │
│                          │
│  하리보 골드베렌 100g    │  ← 상품 컨텍스트
│                          │
│  이 상품 어땠나요?       │
│  ⭐️⭐️⭐️⭐️☆          │  ← Step 1
│                          │
│  이 상품 특징은?         │  ← Step 2 (선택 시 노출)
│  [맛있어요] [달아요]    │
│  [가성비 좋아요]        │
│  [양이 적어요]          │
│  [재구매 의사 있음]     │
│                          │
│  어디서 샀나요? (선택)  │  ← 구매처
│  [GS25 ✓] [CU] [세븐]  │
│                          │
│  한 줄로 남겨주세요      │  ← Step 3 (선택)
│  ┌──────────────────┐   │
│  │ 예: 생각보다 맛있 │   │
│  │ 는데 조금 달아요  │   │
│  └──────────────────┘   │
│                          │
│  [ 리뷰 등록하기 ]       │  ← Step 4
│  10초면 충분해요         │
└─────────────────────────┘
```

### 🚨 절대 원칙

> **한 화면에서 끝낸다. 절대 여러 페이지로 나누지 마라.**

바텀시트 또는 모달 풀스크린 — 둘 다 OK. 핵심은 **스크롤 한 번 이내**로 모든 입력이 끝나야 한다는 것.

## 2. Step-by-Step 설계

### 🟢 Step 1. 별점 (진입 순간)

```
이 상품 어땠나요?

⭐️⭐️⭐️⭐️☆
```

**UX 동작:**
- 별 탭 → 즉시 다음 단계(태그 영역) **자동 확장**
- 별점 미선택 시 제출 버튼 비활성화
- 별점은 **유일한 필수 입력**

**별점별 감정 메시지:**

| 별점 | 메시지 |
|------|--------|
| ⭐ 1 | "별로였나요?" |
| ⭐ 2 | "아쉬웠나요?" |
| ⭐ 3 | "무난했나요?" |
| ⭐ 4 | "괜찮았나요?" |
| ⭐ 5 | "정말 맛있었나요?" |

**목적:** 감정 유도 → 태그/한줄평 작성 이어지기 쉬워짐

---

### 🟡 Step 2. 빠른 선택 (전환율 핵심)

```
이 상품 특징은?

[맛있어요] [달아요] [가성비 좋아요]
[양이 적어요] [재구매 의사 있음]
```

**이게 왜 전환율의 핵심인가:**
- **텍스트 입력 없이 리뷰 완성 가능**
- 클릭만 하면 끝 → 글쓰기 부담 제거
- 이 단계가 없으면 "글 못 쓰겠어" → 이탈

**UX 포인트:**
- 다중 선택 가능 (체크박스)
- 선택 시 즉시 강조 표시
- 태그는 카테고리별로 다를 수 있음 (향후 확장)

**MVP 공통 태그:**

| 태그 | 코드 | 의미 |
|------|------|------|
| 맛있어요 | `taste_good` | 맛 긍정 |
| 별로예요 | `taste_bad` | 맛 부정 |
| 가성비 좋아요 | `value_good` | 가성비 긍정 |
| 가성비 아쉬워요 | `value_bad` | 가성비 부정 |
| 양이 많아요 | `amount_large` | 양 긍정 |
| 양이 적어요 | `amount_small` | 양 부정 |
| 재구매 의사 있음 | `repurchase_yes` | 재구매 의사 |
| 호불호 갈림 | `controversial` | 호불호 |

**주의:** MVP에서는 별점과 일부 태그가 중복될 수 있음 → 괜찮음. 유저가 클릭한 것이 데이터.

---

### 🟣 Step 2.5. 구매처 선택 (선택)

```
어디서 샀나요? (선택)

[GS25 ✓]  [CU]  [세븐일레븐]  [이마트24]  [기타]
```

**정책:**
- **선택 사항** (선물 받은 경우 등 모를 수 있음)
- 단일 선택
- 기본값: 진입한 상품 상세 페이지의 첫 리테일러 (예측)
- 없으면 `null`로 저장

---

### 🔵 Step 3. 한 줄 리뷰 (선택)

```
한 줄로 남겨주세요 (선택)

┌──────────────────────────┐
│                          │
│ 예: 생각보다 맛있는데    │
│ 조금 달아요               │
└──────────────────────────┘
```

**핵심:**
- **필수가 아님** — 비워도 제출 가능
- **플레이스홀더 예시 필수** — "뭘 써야 하지?" 공포 제거
- 최대 200자 정도 (너무 길면 UI로 유도하지 않음)

**플레이스홀더 랜덤 풀:**
- "예: 생각보다 맛있는데 조금 달아요"
- "예: 양이 좀 적어서 아쉬워요"
- "예: 또 사 먹을 것 같아요"
- "예: 친구한테 추천하고 싶어요"

**글자 수 카운터:** 노출하지 않음 (부담 유발)

---

### 🔴 Step 4. 제출

```
┌─────────────────────────┐
│  [ 리뷰 등록하기 ]      │
│                         │
│   10초면 충분해요        │
└─────────────────────────┘
```

**버튼 상태:**
- 별점 없음 → 비활성화 (회색)
- 별점 선택 → 활성화

**보조 메시지:**
- 별점만 선택: "10초면 충분해요"
- 태그 1개 이상 선택: "거의 다 됐어요!"
- 모든 입력 완료: "완벽해요!"

## 3. 전환율 올리는 UX 트릭

### 🔥 트릭 1: "이미 반쯤 쓴 상태 만들기"

별점 선택 직후 태그 영역이 슬라이드 애니메이션으로 펼쳐짐 → **"이미 반쯤 완성됐네, 마저 해볼까"** 심리 유도.

### 🔥 트릭 2: "빈칸 공포 제거"

플레이스홀더 예시 텍스트 필수. 빈 입력창은 공포의 대상.

### 🔥 트릭 3: "선택형 먼저, 입력은 나중"

순서: **별점 → 태그 → 구매처 → 텍스트**
절대 텍스트를 먼저 요구하지 마라.

### 🔥 트릭 4: "완료 보상 느낌"

제출 후 즉시:

```
┌─────────────────────────┐
│                         │
│          🎉             │
│                         │
│   리뷰 작성 완료!        │
│                         │
│  다른 사람들에게          │
│  큰 도움이 되었어요        │
│                         │
│  🏆 First Reviewer      │  ← 첫 리뷰어인 경우
│       뱃지 획득!         │
│                         │
│   [ 확인 ]              │
└─────────────────────────┘
```

### 🔥 트릭 5: 리뷰 길이 제한 없음 (UI로만 유도)

DB 레벨로 최대 2000자 허용, 하지만 UI로는 1~2줄 수준을 유도 (작은 입력창, 글자 수 카운터 없음).

## 5. 로그인 타이밍 (치명적 포인트)

### ❌ 절대 하지 말 것

```
리뷰 쓰기 버튼 클릭
    ↓
로그인 요구
    ↓
80% 이탈 💀
```

### ✅ 올바른 플로우

```
리뷰 쓰기 버튼 클릭
    ↓
리뷰 작성 (비로그인 상태)
    ↓
제출 클릭
    ↓
로그인 (카카오/애플 소셜)
    ↓
로그인 완료 + 리뷰 자동 제출
```

**구현 포인트:**
- 비로그인 상태에서 폼 작성 가능
- 폼 데이터를 **세션 스토리지**에 임시 저장
- 로그인 완료 후 저장된 데이터 자동 제출
- 로그인 실패 시 폼 데이터 유지 (잃지 않음)

## 8. 절대 하면 안 되는 것

| ❌ 안티패턴 | 이유 |
|-------------|------|
| **긴 폼 (맛/가성비/양 점수 각각 입력)** | MVP에서 치명적 — 이탈률 ↑↑↑. 태그로 통합 |
| **필수 텍스트 입력** | 전환율 박살남. 텍스트는 반드시 선택 |
| **페이지 이동** | 한 화면에서 끝내라. 멀티 스텝 UI 금지 |
| **리뷰 작성 전 로그인 강제** | 80% 이탈. 제출 시점에 로그인 |
| **글자 수 카운터** | 부담 유발. 노출 금지 |
| **별점 텍스트 라벨만 있음** | 시각적 자극 부족. 별 아이콘 필수 |
| **확인 팝업 ("정말 등록하시겠어요?")** | 불필요한 마찰. 즉시 제출 |
| **"리뷰 작성" 같은 딱딱한 문구** | "어땠나요?" 같은 대화체 권장 |

## 9. 데이터 모델 영향

### `reviews` 테이블 재확인

MVP 리뷰 폼이 기존 ERD와 어떻게 매핑되는지:

| UI 입력 | DB 필드 | 비고 |
|---------|---------|------|
| 별점 | `rating` | 필수 (1~5) |
| 태그 (맛있어요 등) | **신규: `review_tags` 테이블** | 다중 선택 |
| 구매처 | `retailer_id` | 선택 |
| 한 줄 평 | `body` | 선택 (빈 문자열 허용) |
| 재구매 태그 | `repurchase_intent` | 태그 `repurchase_yes` 선택 시 `true` |

### 🚨 ERD 변경 필요

기존 ERD는 `taste_score`, `value_score`, `amount_score` 숫자 필드였는데,
리뷰 작성 UX가 **태그 기반 경량 폼**으로 변경되면서 재설계 필요:

**Option A (권장): 점수 필드 제거 → 태그 테이블 추가**

```sql
-- review_tags 테이블 신규
CREATE TABLE review_tags (
    id SERIAL PRIMARY KEY,
    review_id INTEGER NOT NULL REFERENCES reviews(id) ON DELETE CASCADE,
    tag_code VARCHAR(30) NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (review_id, tag_code)
);

CREATE INDEX idx_review_tags_tag ON review_tags(tag_code);
CREATE INDEX idx_review_tags_review ON review_tags(review_id);

-- reviews 테이블에서 삭제
-- taste_score, value_score, amount_score 컬럼 제거
-- (repurchase_intent는 유지 — 태그 선택 시 자동 세팅)
```

**Option B: 점수 필드 유지 + 태그 보조**

- 기존 숫자 필드 유지하되 UI에서 수집하지 않음
- 나중에 확장 가능성 열어둠
- 단점: 빈 필드 쌓임, 혼란

**결정 (권장):** **Option A** — MVP 단순성 원칙. 필요하면 나중에 추가.

### `reviews.body` nullable 여부

MVP에서 한 줄 평은 선택이므로 `body`를 `nullable` 또는 빈 문자열 허용으로 변경.

**권장:** `body VARCHAR(2000) NOT NULL DEFAULT ''` (빈 문자열 허용, NOT NULL 유지)

## 10. API 영향

### `POST /api/v1/products/:productId/reviews`

**Request (신규):**
```json
{
  "retailer_id": 1,
  "rating": 4,
  "body": "생각보다 맛있는데 조금 달아요",
  "tags": ["taste_good", "taste_sweet", "value_good", "repurchase_yes"]
}
```

**검증:**
- `rating` 필수 (1~5)
- `body` 선택 (빈 문자열 허용)
- `tags` 선택 (배열, 허용된 `tag_code`만)
- `retailer_id` 선택 (nullable)
- `repurchase_intent`는 `tags`에 `repurchase_yes` 포함 여부로 자동 결정

**응답 (신규 — 성공 시 뱃지 정보 포함):**
```json
{
  "review": {
    "id": 501,
    "rating": 4,
    "body": "생각보다 맛있는데 조금 달아요",
    "tags": ["taste_good", "taste_sweet", "value_good", "repurchase_yes"],
    "repurchase_intent": true,
    "created_at": "2026-04-10T10:30:00Z"
  },
  "awarded_badges": [
    {
      "code": "first_reviewer",
      "name": "첫 리뷰어",
      "icon_url": "..."
    }
  ]
}
```

클라이언트는 `awarded_badges`가 비어있지 않으면 완료 화면에서 뱃지 획득 애니메이션 표시.
